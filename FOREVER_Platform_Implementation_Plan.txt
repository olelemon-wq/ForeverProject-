1. ARCHITECTURE (สถาปัตยกรรมระบบ)
--------------------------------------------------------------------------------
สถาปัตยกรรมของ FOREVER ได้รับการออกแบบให้เป็น Single Next.js Monolith ที่ใช้เทคนิค Route Groups 
เพื่อแบ่งพื้นที่ใช้งาน (Surfaces) ทั้ง 4 ส่วน บนฐานข้อมูลและ Backend เดียวกัน ช่วยให้ดูแลรักษาง่ายและแชร์ Component ร่วมกันได้สะดวก

1.1 การจัดโครงสร้าง Routing (Route Groups ใน Next.js App Router)
   * app/(marketing)/     -> หน้าขาย, Pricing, FAQs, ตัวอย่างเว็บ (www.forever.co.th)
   * app/(public)/[slug]/  -> เว็บไซต์ความทรงจำของผู้ล่วงลับ (forever.co.th/{slug})
   * app/manage/           -> ระบบจัดการฝั่ง Webmaster (forever.co.th/manage)
   * app/admin/            -> ระบบจัดการฝั่งผู้ดูแลระบบสูงสุด (forever.co.th/admin)
   * app/(auth)/           -> หน้าล็อกอินยืนยันตัวตนด้วย SMS OTP เท่านั้น

1.2 เทคโนโลยีหลัก (Tech Stack)
   * Frontend: Next.js 15+ (App Router), React 19
   * Styling & Design System: Tailwind CSS v4 (เน้นการใช้ CSS variables เพื่อทำ runtime theming)
   * UI components: shadcn/ui (สำหรับ Backoffice, Admin และ Marketing)
   * ORM & Database: Prisma ORM เชื่อมต่อกับ PostgreSQL
   * Media Storage: Cloud Object Storage (S3 Compatible เช่น Cloudflare R2 / AWS S3)
   * Task Queue: Redis + BullMQ สำหรับการรันงานหนักในพื้นหลัง (Background Jobs)

1.3 การจัดการ Multi-Tenant (URL & Subdomain Rewrite)
   * ออกแบบ Tenancy Strategy แบบ Shared Database & Shared Process (Logical Isolation)
   * ใช้ Next.js Middleware ในการตรวจสอบ Request:
     - หากเข้ามาด้วย {slug}.forever.co.th จะทำการ Internal Rewrite (ไม่ redirect) 
       ไปที่ /(public)/[slug] เพื่อให้ผู้ใช้งานเห็นหน้าเว็บเหมือนใช้งานผ่าน Subdomain ตัวเอง
     - Middleware จะทำการ Resolve Tenant ID จาก DB และฝังไว้ใน Request Context / Header
     - กำหนด Reserved Keywords (เช่น admin, manage, api, payment, www, test) ในระดับ Middleware 
       เพื่อป้องกันไม่ให้ผู้ใช้ตั้งชื่อ Slug ซ้ำกับเส้นทางระบบ

1.4 Customer Theme Engine (ระบบธีมของลูกค้า)
   * แยกขาดจาก UI System (shadcn) ของระบบจัดการ
   * เมื่อผู้ใช้ปรับแต่งธีม (จากตัวเลือก 10 ธีมหลัก) ระบบจะบันทึกค่าลงใน Database (ตาราง Tenant)
   * หน้าเว็บฝั่ง Public จะอ่านค่าจาก Database แล้วแปลงเป็น CSS Variables ฝังไว้ใน Layout 
     (เช่น --theme-primary, --theme-secondary, --theme-font) ทำให้หน้าเว็บเปลี่ยนสีและฟอนต์ได้ทันที 
     โดยไม่ต้องทำการ Rebuild หน้าเว็บใหม่

2. SCALABILITY (การรองรับการขยายตัว)
--------------------------------------------------------------------------------
เป้าหมายธุรกิจ: ปีที่ 1 = 100 เว็บ, ปีที่ 2 = 500 เว็บ, ปีที่ 3 = 1,000 เว็บ
การออกแบบระบบต้องคำนึงถึงการเติบโตของข้อมูลสื่อ (รูปภาพ/วิดีโอ) และการจราจรทางเครือข่าย

2.1 Stateless Application Servers
   * เซิร์ฟเวอร์ Next.js จะต้องไม่มีการเก็บ State หรือไฟล์ใดๆ ไว้ใน Local Disk
   * เซสชันการล็อกอินและไฟล์ข้อมูลทั้งหมดจะดึงจาก Database และ Cloud Storage เท่านั้น
   * ช่วยให้สามารถทำ Auto-scaling (ขยายตัวตามปริมาณผู้ใช้งาน) ได้ทันทีบน Cloud Run, AWS ECS หรือ Vercel

2.2 Database Scaling
   * สร้าง Database Indexes บนฟิลด์ที่มีการค้นหาบ่อย: (slug), (tenant_id), (expired_at), (status)
   * ใช้ Database Connection Pooler (เช่น PgBouncer หรือ Prisma Accelerate) 
     เพื่อจัดการกับพูลการเชื่อมต่อของเซิร์ฟเวอร์แบบ Serverless ป้องกันปัญหา Connection เต็ม
   * ออกแบบสำหรับอนาคต: หากระบบขยายตัว สามารถทำ Read Replicas สำหรับดึงข้อมูลหน้า Public 
     ซึ่งเป็น Read-heavy (คนเข้ามาเปิดดูคลังรูปภาพ/เขียนคำไว้อาลัย) ออกจากฐานข้อมูลหลักที่เป็น Write-heavy ได้

2.3 Asynchronous Storage & Queue System
   * การอัปโหลดไฟล์ขนาดใหญ่ (วิดีโอ MP4, หนังสือ Ebook, ไฟล์เสียง MP3) จะไม่ผ่าน App Server ตรงๆ 
     แต่ Frontend จะขอ Presigned URL เพื่ออัปโหลดไปยัง Cloud Storage โดยตรง
   * ใช้ BullMQ (รันบน Redis) เพื่อประมวลผลงานที่เป็นภาระหนักแบบ Asynchronous:
     - การลดขนาดรูปภาพและสร้าง Thumbnail อัตโนมัติ (Image Optimization)
     - การประมวลผลวิดีโอ MP4 และสร้างภาพตัวอย่างวิดีโอ (Video Transcoding & Thumbnail)
     - การรวบรวมไฟล์ทั้งหมดและบีบอัดเป็น ZIP ตอนส่งออกข้อมูล (Export ZIP - BR024)
     - การจัดตารางเวลาส่ง SMS แจ้งเตือนก่อนหมดอายุสมาชิก (Expiry Reminder Queue)

3. SECURITY (ความปลอดภัยและการป้องกันสแปม)
--------------------------------------------------------------------------------
3.1 Tenant Data Isolation
   * ป้องกันการรั่วไหลของข้อมูลข้าม Tenant (ข้ามหน้าเว็บผู้ล่วงลับ) ในระดับ Query 
     โดยการบังคับใส่เงื่อนไข `where tenant_id = current_tenant_id` ในทุกๆ Query
   * ตรวจสอบสิทธิ์ของ Webmaster (ทั้ง Main และ Co-Webmaster) ทุกครั้งในระบบหลังบ้าน 
     โดยใช้ Middleware เช็กความสัมพันธ์ระหว่าง User Phone และ Website ID ใน Session 
     ก่อนอนุญาตให้เข้าถึง API /manage

3.2 SMS OTP & Authentications
   * ล็อกอินด้วย OTP ผ่านเบอร์มือถือเท่านั้น ไม่มีรหัสผ่าน เพื่อลดความยุ่งยากให้กลุ่มผู้สูงอายุ (40-70 ปี)
   * ออกแบบระบบป้องกันการส่ง OTP รัวๆ (SMS Fraud / OTP Throttling):
     - จำกัดการขอ OTP ได้สูงสุด 1 ครั้งต่อ 60 วินาทีต่อเบอร์โทรศัพท์
     - จำกัดสูงสุด 5 ครั้งต่อ 1 ชั่วโมง
     - ล็อกเบอร์โทรศัพท์ชั่วคราว 24 ชั่วโมง หากป้อน OTP ผิดติดต่อกันเกิน 5 ครั้ง
   * ระบบ Trusted Device: บันทึกข้อมูลเบราว์เซอร์และอุปกรณ์ที่เคยล็อกอินในฐานข้อมูล 
     ผู้ใช้สามารถตรวจสอบรายชื่ออุปกรณ์และสั่ง Remote Logout (ลบอุปกรณ์ที่เชื่อถือได้) ได้ตลอดเวลา

3.3 Payment Integration Security
   * ระบบชำระเงินรองรับเฉพาะ PromptPay QR (Dynamic QR) และปลดล็อกระบบผ่าน Webhook Callback เท่านั้น (BR018)
   * ความปลอดภัยฝั่ง Webhook API Endpoint:
     - ตรวจสอบ Signature/Checksum (เช่น SHA-256 HMAC) ที่ส่งมาจาก Payment Gateway 
       เพื่อตรวจสอบว่าเป็นข้อมูลจากธนาคารจริง ป้องกันการเจาะระบบเพื่อเปิดใช้งานฟรี
     - ป้องกัน Idempotency: ระบบต้องเช็กก่อนว่าเลขที่ธุรกรรม (Transaction ID) 
       นี้เคยถูกประมวลผลไปแล้วหรือยัง เพื่อไม่ให้เกิดการอัปเดตวันหมดอายุซ้ำซ้อน
     - มีระบบตรวจสอบย้อนกลับ (Audit Log) ทุกครั้งที่มีรายการ Callback เข้ามา

3.4 Spam & Abuse Prevention (การรักษาระดับความปลอดภัยภายนอก)
   * ป้องกันสแปมคำไว้อาลัย (Condolence System) ด้วยการผูก Cloudflare Turnstile หรือ Google reCAPTCHA v3 
     ในหน้ารับชมสาธารณะ
   * คำไว้อาลัยเริ่มต้นจะไม่อนุมัติให้ออกอากาศโดยอัตโนมัติ (Default is Pending approval) 
     จนกว่า Webmaster จะกดอนุมัติ (ยกเว้นเว็บบอร์ดที่ Webmaster เลือกปิดฟีเจอร์อนุมัติคำไว้อาลัยเอง - BR027)
   * การจำกัด Storage: เมื่อใช้งานพื้นที่ครบ 100% (คำนวณจากขนาดไฟล์ภาพ, วิดีโอ, เสียง, เอกสารต้นฉบับ ไม่นับ Thumbnail) 
     ระบบจะล็อกการอัปโหลดเพิ่มทันที แต่ผู้เข้าชมทั่วไปยังคงเข้าดูและส่งคำไว้อาลัยข้อความได้ตามปกติ (BR016)

3.5 Audit Logs & Backups
   * มีระบบบันทึกประวัติการกระทำสำคัญ (Audit Logs - BR036): การล็อกอิน, การชำระเงิน, การเผยแพร่เนื้อหา, 
     การลบข้อมูล และการซื้อพื้นที่เพิ่ม โดยเก็บบันทึกประวัติไว้เป็นเวลาอย่างน้อย 2 ปี
   * ระบบสำรองข้อมูล (Daily Backup - BR039): สำรองข้อมูลฐานข้อมูลอัตโนมัติทุกวัน และมีระยะเวลาเก็บรักษา (Retention) 30 วัน

4. PERFORMANCE (ประสิทธิภาพและการประมวลผล)
--------------------------------------------------------------------------------
4.1 Edge Caching & Content Delivery Network (CDN)
   * ใช้บริการ CDN (เช่น Cloudflare) ตั้งแต่วันแรก เพื่อทำ Cache หน้า Public Memorial 
     ลดการร้องขอข้อมูลไปยังต้นทาง (Origin Server)
   * สำหรับหน้าความทรงจำของผู้ล่วงลับ (/(public)/[slug]):
     - ใช้ Next.js Incremental Static Regeneration (ISR) ในการเรนเดอร์หน้าเว็บแบบ Static HTML 
       ทำให้เว็บโหลดเร็วสูงสุด (ต่ำกว่า 1.5 วินาที)
     - เมื่อ Webmaster แก้ไขข้อมูลหลังบ้าน จะมีคำสั่ง On-Demand Revalidation 
       เพื่อล้าง Cache หน้าของคนคนนั้นบน Edge Node ทันที

4.2 Image Optimization
   * บังคับใช้รูปภาพผ่าน Component `next/image` ใน Next.js ซึ่งจะแปลงรูปภาพเป็นฟอร์แมตยุคใหม่ (WEBP หรือ AVIF) 
     และทำการย่อขนาดรูปตามหน้าจอผู้ใช้อัตโนมัติ (Responsive images via Srcset)
   * จัดการระบบอัปโหลดแบบ Lazy loading สำหรับคลังภาพขนาดใหญ่ (Gallery) ป้องกันไม่ให้เบราว์เซอร์ดาวน์โหลดไฟล์รูปพร้อมกันทีเดียว

4.3 Video & Audio Delivery
   * วิดีโอ MP4 ที่อัปโหลดจะถูกเก็บที่ Object Storage และให้บริการผ่าน CDN 
     โดยจะใช้วิธีเล่นแบบ Byte-range request (Streaming chunks) 
     เพื่อประหยัดแบนด์วิดท์ และทำให้ผู้ใช้ดูวิดีโอได้ทันทีโดยไม่ต้องรอโหลดไฟล์เต็ม
   * เพลงรำลึกที่เล่นในพื้นหลัง (Audio player) จะต้องถูกบีบอัดให้อยู่ในมาตรฐาน 128kbps MP3 
     เพื่อคุณภาพเสียงที่ดีแต่ไฟล์มีขนาดเล็กไม่เป็นภาระของเครือข่ายมือถือ

4.4 Database Query Optimization
   * ใช้ Pagination (Cursor-based) ในหน้าแสดงรายการที่ขยายตัวเรื่อยๆ เช่น คลังภาพ (Gallery) และคำไว้อาลัย (Condolences)
   * เลี่ยงการทำ SQL `JOIN` หลายชั้นในระบบคิวรี่ ให้เน้นเก็บข้อมูลแบบ Normalized Schema และนำข้อมูลมารวมกันใน Memory ชั้นแอปพลิเคชัน

5. SEO (การค้นหาและการจัดอันดับ)
--------------------------------------------------------------------------------
5.1 Dynamic Meta Tags Generation
   * ใช้ฟังก์ชัน `generateMetadata` ใน Next.js สำหรับเรนเดอร์หน้าสาธารณะ (public memorial) 
     โดยจะสร้าง Tag เหล่านี้ในระดับ Server-Side เสมอเพื่อให้ Search Engine Crawlers (เช่น Googlebot) 
     มาเก็บข้อมูลและนำไปแสดงผลได้อย่างถูกต้อง
   * โครงสร้าง Meta Tags:
     - `<title>`: "รำลึกถึง [ชื่อผู้ล่วงลับ] - [ความสัมพันธ์] | FOREVER"
     - `<meta name="description">`: "ร่วมรำลึก ร่วมจุดเทียน และเขียนคำไว้อาลัยแด่ [ชื่อผู้ล่วงลับ] เพื่อบันทึกความทรงจำอันทรงคุณค่าให้อยู่ตลอดไป"
     - Open Graph (og:title, og:description, og:image): ดึงรูปภาพโปรไฟล์หลักของผู้ล่วงลับไปเป็น og:image เสมอ 
       เพื่อให้เวลานำลิงก์ไปแชร์ใน Facebook หรือ LINE จะแสดงภาพตัวอย่างที่สวยงาม

5.2 Canonical URLs & Subdomain Management
   * จากข้อกำหนด BR034: ระบบต้องใช้ URL Path เป็นหลัก ส่วน Subdomain เป็นทางเลือกชี้มาหน้าเดียวกัน
   * เพื่อป้องกัน Google ตรวจพบ "เนื้อหาซ้ำซ้อน" (Duplicate Content Penalty) 
     ทุกหน้าสาธารณะที่ถูกเปิดเข้าผ่าน {slug}.forever.co.th จะต้องมี Tag Canonical 
     ชี้ไปที่ URL หลักเสมอ คือ `<link rel="canonical" href="https://forever.co.th/{slug}" />`

5.3 Structured Data (JSON-LD Schema)
   * ฝังโครงสร้างข้อมูล JSON-LD สำหรับโปรไฟล์บุคคลผู้ล่วงลับลงในหน้า HTML ทุกหน้า เพื่อให้กูเกิลนำไปแสดงผลแบบ Rich Snippet
   * ตัวอย่าง Schema.org Object ที่เลือกใช้:
     - `Person` (พร้อมคุณสมบัติ `deathDate`, `birthDate`, `birthPlace`, `deathPlace`, `description`)

5.4 Sitemaps & robots.txt
   * ระบบจะสร้างไฟล์ sitemap.xml แบบ Dynamic ซึ่งจะดึงรายชื่อ URL ของเว็บไซต์ผู้ล่วงลับ
     ที่ยังไม่หมดอายุสมาชิกและตั้งสถานะเป็น Public เท่านั้นมาใส่ในไฟล์ sitemap
   * กำหนด robots.txt เพื่อปิดกั้นการเข้าถึงหน้า /manage, /admin, /auth และหน้าเว็บไซต์ที่ตั้งค่าความเป็นส่วนตัวไว้

6. BACKEND & FRONTEND IMPLEMENTATION (การพัฒนาระบบหลังบ้านและหน้าบ้าน)
--------------------------------------------------------------------------------
6.1 Frontend Component Design
   * ใช้ Tailwind CSS v4 สำหรับควบคุม Variable CSS Theme ทั้งหมด 
     ทำให้ผู้ใช้งานสามารถปรับแต่ง Dynamic Theme Engine ได้โดยไม่กระทบความเร็วโหลด
   * ใช้ React Hooks และ State Management ในหน้า Manage เพื่ออำนวยความสะดวกในฟังก์ชัน:
     - Drag & Drop Menu Builder (จัดการลำดับสไลด์ หน้าเมนู ความเห็น และปิดหน้านั้นๆ เมื่อไม่มีข้อมูล - BR026)
     - Multi-file Drag & Drop Uploader พร้อมแถบแสดงเปอร์เซ็นต์ความคืบหน้าการอัปโหลดแต่ละไฟล์

6.2 Backend Architecture & Services
   * แบ่งระบบการทำงาน (Domain Services) หลังบ้านออกเป็นโมดูลย่อยชัดเจน:
     - Tenant Service: สำหรับเช็ก Slug, Reserved Words และดึงตั้งค่าของแต่ละเว็บ
     - Auth Service: ระบบ OTP Generation, SMS Dispatch, และ Session Token Management
     - Storage Service: คำนวณปริมาณพื้นที่ใช้งานของลูกค้ารายตัว (BR014), จัดการ Presigned URL
     - Subscription Service: ตรวจสอบสถานะการใช้งานและจัดการ Lifecycle (Active -> Read-Only -> Suspend -> Archive - BR021)
     - Payment Webhook Service: รับสัญญาณจากธนาคาร, ตรวจสอบลายเซ็นดิจิทัล, บันทึกการชำระเงิน, ออกใบเสร็จ (Receipt) 
       และใบกำกับภาษี (Tax Invoice) อัตโนมัติ (BR019)

6.3 API Endpoint Contract (ตัวอย่างสัญญา API สำคัญ)
   * `POST /api/auth/otp/request` -> รับเบอร์โทรศัพท์เพื่อส่ง OTP ทาง SMS
   * `POST /api/auth/otp/verify` -> ยืนยันรหัส OTP เพื่อแลก Session/Cookie
   * `POST /api/tenant/create` -> ตรวจสอบความถูกต้องของสเปกสร้างเว็บ, หักเงิน และสั่งสร้าง Tenant ในระบบ
   * `POST /api/payment/callback` -> Endpoint สำหรับผู้ให้บริการชำระเงินส่งสัญญาณ Webhook
   * `GET /api/media/upload-url` -> ขอ Presigned URL จาก Object Storage เพื่ออัปโหลดไฟล์ตรงจากเบราว์เซอร์

7. MOBILE SUPPORT (การรองรับอุปกรณ์พกพา - MOBILE FIRST)
--------------------------------------------------------------------------------
กลุ่มเป้าหมายหลักมีอายุอยู่ในช่วง 40-70 ปี และใช้งานผ่านอุปกรณ์โทรศัพท์มือถือเป็นหลัก (Mobile First - BR044)

7.1 Touch Targets & Input Fields
   * ออกแบบขนาดปุ่มกด ลิงก์ และองค์ประกอบที่ต้องสัมผัสให้มีขนาดไม่ต่ำกว่า 48x48 พิกเซล เพื่อให้ผู้สูงอายุกดได้ง่าย
   * ตั้งค่าฟอนต์ของช่องกรอกข้อมูล (Input) ในมือถือให้มีขนาดไม่ต่ำกว่า 16 พิกเซล 
     เพื่อป้องกันไม่ให้เบราว์เซอร์บนระบบปฏิบัติการ iOS (Safari) 
     ทำการซูมหน้าจออัตโนมัติขณะพิมพ์ ซึ่งจะทำให้ผู้สูงอายุสับสนและใช้งานยาก

7.2 Responsive Layout
   * ออกแบบหน้าเว็บด้วยระบบ Grid และ Flexbox ที่ปรับเปลี่ยนรูปแบบอัตโนมัติ (Fluid responsive layout) 
     หน้าเว็บต้องไม่หลุดขอบและไม่มีสกรอลบาร์ในแนวนอน (Horizontal Scroll) ในทุกความละเอียดหน้าจอมือถือ
   * ใช้ปุ่มนำทางแบบลอยตัว (Floating Action Button - FAB) สำหรับคำสั่งยอดนิยม เช่น "เขียนคำไว้อาลัย" หรือ "ส่งภาพรำลึก"

7.3 Optimization for Low Network & Old Devices
   * ผู้ใช้อาจเปิดลิงก์ผ่าน 3G/4G ขณะอยู่ที่งานฌาปนกิจ หน้าเว็บฝั่ง Public จึงต้องถูกรีดประสิทธิภาพสูงสุด:
     - ตัดไลบรารี JS ขนาดใหญ่ที่ไม่จำเป็นออก
     - บีบอัดขนาดรูปภาพและวิดีโอตัวอย่าง (Thumbnail) ให้เหมาะกับจอโทรศัพท์
     - รองรับการทำงานแบบ Offline หรือเชื่อมต่ออินเทอร์เน็ตได้ช้า ด้วยการแจ้งเตือนที่เข้าใจง่าย

7.4 QR Code Mobile Integration
   * เว็บไซต์แต่ละหน้าจะได้รับ QR Code ประจำตัวถาวร (BR032) สำหรับนำไปพิมพ์ลงในหนังสืองานศพหรือการ์ดเชิญ
   * หน้ารวมความทรงจำต้องออกแบบให้มีปุ่มสำหรับ "ดาวน์โหลด QR Code" ลงในเครื่องสมาร์ทโฟนได้ง่าย 
     โดยจะส่งไฟล์ที่มีคุณภาพสูง (PNG, SVG, PDF) ไปให้ผู้ใช้งาน เพื่อความชัดเจนในการนำไปพิมพ์ต่อ

[ ] Task 4.4: พัฒนาระบบตรวจสอบ Lifecycle ของสมาชิกอัตโนมัติ (Subscription Expiry Cron Job)

      - หมดอายุ 1-30 วัน: สถานะเป็น Read-Only (แก้ไขข้อมูลไม่ได้ แต่หน้าเว็บยังเปิดดูและส่งคำไว้อาลัยได้)
      - หมดอายุ 31-60 วัน: สถานะเป็น Suspend (ปิดกั้นไม่ให้คนทั่วไปเข้าใช้งานชั่วคราว)
      - หมดอายุ 61-180 วัน: สถานะเป็น Archive (ซ่อนข้อมูลจากหน้าสืบค้นสาธารณะ ข้อมูลไม่ถูกลบ)
      - หมดอายุมากกว่า 180 วัน: เสนอให้ Admin ตรวจสอบเพื่อดำเนินการต่อ (ไม่มีการลบไฟล์อัตโนมัติ)
